\`# Tor exit node API — the 2026 developer guide to blocking Tor users at signup

Tor exit node API is the niche but important category for fraud prevention. The honest answer: detecting Tor users at signup is technically trivial (every vendor does it correctly), but the ethical question — whether you SHOULD block Tor users — is where most SaaS companies get it wrong.

This post covers both: how Tor exit node detection actually works, how to implement it at signup, and the right way to handle the journalist / activist traffic that legitimate Tor users represent.

## How Tor exit node detection works

The Tor network publishes a list of every exit node (the final hop before traffic reaches the destination website) every hour. This list is publicly available at:

- **https://check.torproject.org/torbulkexitlist** (Tor Project's official list, updated hourly)
- **https://www.dan.me.uk/torlist/** (community-maintained, updated daily)
- **MaxMind**, **IPQualityScore**, and **SignupDoggy** maintain their own (often larger) lists

A Tor exit node API is just a wrapper around one of these lists. You send an IP, the API tells you whether it's a Tor exit node. That's the whole technical implementation.

## The ethics of blocking Tor

Most SaaS companies default to hard-blocking all Tor traffic. This is wrong for three reasons:

1. **Journalists and activists**. A journalist investigating a corrupt government in their own country uses Tor to protect their source's identity. If they sign up for your tool to research your competitor, blocking them loses you a legitimate user who needs your tool.
2. **People in censored countries**. Users in China, Iran, Russia, and similar countries use Tor to access the global internet. If they want to sign up for a tool that's blocked in their country, blocking them at signup is a UX failure.
3. **Security-conscious users**. Some users just prefer Tor. They might be paranoid, but they're not fraudsters.

The right approach: hard-block Tor for high-value accounts (anything tied to money, identity, or sensitive data), but soft-block Tor (require email confirmation before account activation) for low-value accounts (newsletters, free-tier SaaS).

## How to implement Tor exit node detection at signup

Three approaches, from simplest to most robust:

### Approach 1: free list download (smallest scale)

    import fs from 'fs';
    import { createHash } from 'crypto';

    // Download the Tor Project list hourly
    async function refreshTorList() {
      const res = await fetch('https://check.torproject.org/torbulkexitlist');
      const text = await res.text();
      const ips = text.split('\n').filter(l => /^\d+\.\d+\.\d+\.\d+$/.test(l));
      fs.writeFileSync('/tmp/tor-exits.txt', ips.join('\n'));
    }

    // Check if an IP is a Tor exit
    function isTorExit(ip) {
      const list = fs.readFileSync('/tmp/tor-exits.txt', 'utf8').split('\n');
      return list.includes(ip);
    }

This works but requires you to maintain the list yourself (hourly refresh), and it doesn't scale beyond a few hundred requests/second.

## The right answer for most SaaS companies

**Soft-block Tor**: detect it, log it, require email confirmation before account activation. Hard-block for high-value accounts (anything tied to money, identity, or sensitive data).

This catches 99%+ of Tor-based fraud (the fraudsters want a one-click signup; they don't want to verify an email) while letting the 1% of legitimate Tor users through.

Implementation:

    if (ipCategory === 'tor') {

The account is created but cannot be used until the user verifies their email. This is friction for the fraudster (they need a working email address they control, which defeats most Tor-based fraud) but transparent for the legitimate user.
